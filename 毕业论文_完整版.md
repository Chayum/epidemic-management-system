## 摘要

随着全球公共卫生事件的频发，疫情防控已成为世界各国面临的重大挑战。在疫情防控工作中，医疗物资的及时供应与高效调配是保障人民生命安全和身体健康的关键环节。传统的物资管理模式存在信息孤岛、响应滞后、资源调配不合理等问题，难以满足突发公共卫生事件下快速响应的需求。因此，开发一套高效、智能的疫情防控物资调度管理系统具有重要的现实意义。

本文针对疫情防控物资调度管理的实际需求，设计并实现了一套基于微服务架构的物资调度管理系统。系统采用Spring Cloud Alibaba作为微服务框架，结合Spring Boot、MyBatis-Plus、JWT、Redis、RabbitMQ等技术栈，实现了用户管理、物资管理、库存管理、申请管理、捐赠管理、疫情信息发布和数据统计分析等核心功能。系统采用前后端分离架构，前端基于Vue 3和Element Plus构建，后端通过网关服务进行统一路由管理，各业务服务独立部署，实现了服务的高内聚低耦合。

在技术实现方面，本文重点研究了以下关键技术点：（1）基于JWT+Redis的双重验证机制，实现了用户身份的安全认证与单点登录；（2）利用Redisson分布式锁技术，保障了高并发场景下库存操作的原子性和数据一致性；（3）采用RabbitMQ消息队列实现异步日志处理，提升系统响应性能；（4）设计多级缓存策略，有效降低了数据库访问压力，提高了系统吞吐量。

经过系统测试和实际应用验证，该系统能够有效支撑疫情防控期间的物资调度工作，实现了物资信息的实时共享、申请流程的在线审批、库存状态的动态监控和数据统计的自动化分析。系统具有良好的可扩展性、可维护性和稳定性，为提升疫情防控物资管理水平提供了有力的技术支撑。

**关键词：** 疫情防控；物资调度；微服务架构；Spring Cloud；分布式系统

## Abstract

With the frequent occurrence of global public health events, epidemic prevention and control have become major challenges facing countries around the world. In epidemic prevention and control work, the timely supply and efficient allocation of medical materials are crucial for safeguarding people's lives and health. Traditional material management models suffer from information silos, delayed responses, and unreasonable resource allocation, making it difficult to meet the needs for rapid response during sudden public health events. Therefore, developing an efficient and intelligent epidemic prevention material scheduling and management system has significant practical importance.

This paper addresses the practical needs of epidemic prevention material scheduling and management by designing and implementing a material scheduling management system based on microservices architecture. The system adopts Spring Cloud Alibaba as the microservices framework, combined with Spring Boot, MyBatis-Plus, JWT, Redis, RabbitMQ, and other technology stacks, implementing core functions such as user management, material management, inventory management, application management, donation management, epidemic information publishing, and data statistical analysis. The system adopts a front-end and back-end separation architecture, with the front-end built on Vue 3 and Element Plus, and the back-end managed through a gateway service for unified routing. Each business service is deployed independently, achieving high cohesion and low coupling of services.

In terms of technical implementation, this paper focuses on the following key technical points: (1) A dual authentication mechanism based on JWT+Redis, achieving secure user authentication and single sign-on; (2) Utilizing Redisson distributed lock technology to ensure atomicity and data consistency of inventory operations in high-concurrency scenarios; (3) Adopting RabbitMQ message queues for asynchronous log processing to improve system response performance; (4) Designing a multi-level caching strategy to effectively reduce database access pressure and improve system throughput.

Through system testing and practical application verification, this system can effectively support material scheduling work during epidemic prevention and control, achieving real-time sharing of material information, online approval of application processes, dynamic monitoring of inventory status, and automated analysis of data statistics. The system has good scalability, maintainability, and stability, providing strong technical support for improving the management level of epidemic prevention and control materials.

**Keywords:** Epidemic Prevention and Control; Material Scheduling; Microservices Architecture; Spring Cloud; Distributed System

## 第1章 引言

### 1.1 研究背景

21世纪以来，全球范围内频繁爆发各类重大传染病疫情，从2003年的非典型肺炎（SARS）到2009年的甲型H1N1流感，从2014年的埃博拉病毒到2020年初爆发的新型冠状病毒肺炎（COVID-19），每一次疫情都对人类社会造成了深远的影响。特别是COVID-19疫情，其传播速度快、感染范围广、防控难度大，给全球公共卫生体系带来了前所未有的挑战。

在疫情防控工作中，医疗物资的供应保障是决定防控成效的关键因素之一。口罩、防护服、护目镜、消毒液、呼吸机等医疗物资不仅是医护人员开展救治工作的必备装备，也是普通民众进行自我防护的重要工具。然而，在疫情爆发初期，由于需求激增、供应链中断、信息不对称等原因，许多地区出现了医疗物资严重短缺的情况，严重影响了疫情防控工作的开展。

传统的物资管理模式主要依赖人工记录和纸质文档，存在信息传递滞后、数据更新不及时、资源调配效率低下等问题。在常态情况下，这种管理模式或许能够维持基本运转，但在突发公共卫生事件的紧急状态下，其弊端暴露无遗：一方面，各级医疗机构和物资管理部门之间缺乏有效的信息共享机制，形成了一个个"信息孤岛"，导致物资需求无法及时上报、物资库存无法实时掌握；另一方面，物资申请和审批流程繁琐，往往需要经过多级审批，耗时较长，无法满足紧急调度的需求；此外，由于缺乏科学的预测和分析工具，物资的采购、储备和分配往往缺乏依据，容易造成资源浪费或供应不足。

随着信息技术的快速发展，互联网、大数据、云计算、人工智能等新技术为物资管理模式的创新提供了新的可能。通过构建数字化的物资调度管理系统，可以实现物资信息的实时采集、传输和处理，打破信息孤岛，提高信息透明度；通过智能化的分析和预测，可以科学制定采购计划和调配方案，提高资源利用效率；通过在线化的业务流程，可以简化申请审批环节，提高响应速度。因此，研究和开发一套适用于疫情防控场景的物资调度管理系统，对于提升我国公共卫生事件应急响应能力具有重要的现实意义。

### 1.2 研究意义

本课题的研究意义主要体现在以下几个方面：

**（1）理论意义**

本研究将微服务架构、分布式系统、消息队列等先进技术应用于疫情防控物资管理领域，探索信息技术与公共卫生管理的深度融合，丰富了应急管理信息化的理论体系。通过对系统架构设计、数据一致性保障、高并发处理等关键技术的研究，为类似系统的开发提供了理论参考和技术借鉴。

**（2）实践意义**

从实践角度来看，本系统的开发和应用具有以下重要意义：

第一，提升物资调度效率。通过系统化的信息管理，实现物资需求的快速上报、申请的在线审批、库存的实时监控，大大缩短了物资调度的响应时间，提高了应急处置能力。

第二，优化资源配置。系统提供的数据统计和分析功能，可以帮助管理者全面了解物资的供需状况、消耗趋势和库存分布，为科学决策提供数据支撑，避免资源的浪费和短缺。

第三，增强信息透明度。系统实现了物资信息的公开透明，各级用户可以实时查询物资的库存、流向和使用情况，有利于加强监督，防止腐败现象的发生。

第四，积累数据资产。系统运行过程中积累的大量业务数据，可以用于后续的疫情分析和预测研究，为完善公共卫生应急管理体系提供数据基础。

**（3）社会意义**

本系统的开发和推广使用，对于保障人民群众生命安全和身体健康、维护社会稳定具有积极的社会意义。通过提高物资调度的效率和精准度，可以确保医疗物资及时送达一线，为疫情防控赢得宝贵时间；通过规范化的流程管理，可以减少人为因素的干扰，提高工作的公平性和公正性；通过信息的公开透明，可以增强公众对疫情防控工作的信心和支持。

### 1.3 国内外研究现状

#### 1.3.1 国外研究现状

在发达国家，公共卫生应急管理体系建设起步较早，物资调度管理信息化程度相对较高。

美国建立了较为完善的应急物资管理体系，联邦应急管理署（FEMA）负责统筹协调全国的应急物资储备和调配工作。美国开发了多种应急管理软件系统，如EPLS（Excluded Parties List System）用于管理物资供应商信息，FPDS（Federal Procurement Data System）用于追踪政府采购数据。在COVID-19疫情期间，美国还开发了多款用于追踪疫情和物资需求的应用程序。

日本由于其特殊的地理环境和频发的自然灾害，建立了成熟的灾害物资管理体系。日本开发了<D-SUPPORT>系统，实现了灾害发生时物资需求的快速收集、救援物资的高效调配和物流信息的实时追踪。该系统采用了GIS技术和移动应用技术，提高了物资调度的精准度和效率。

欧洲各国在欧盟框架下建立了联合应急物资储备和调配机制。欧盟开发了HERA（Health Emergency Preparedness and Response Authority）系统，用于协调成员国之间的医疗物资采购和分配。该系统通过统一的平台实现了需求汇总、资源匹配和物流协调，提高了欧盟整体的应急响应能力。

从技术架构来看，国外的应急管理系统普遍采用了分布式架构、云计算技术和大数据分析技术，注重系统的可扩展性和容错性。在数据安全方面，注重隐私保护和访问控制，普遍采用了身份认证、加密传输等安全措施。

#### 1.3.2 国内研究现状

近年来，我国在公共卫生信息化建设方面取得了显著进展，特别是在COVID-19疫情之后，各级政府和社会各界对公共卫生应急管理的重视程度大大提高。

在国家层面，国家卫生健康委建立了全国公共卫生应急物资储备信息平台，实现了中央和地方物资储备信息的汇总和共享。工业和信息化部开发了重点医疗物资保障调度平台，对重点医疗物资的生产、流通、储备进行全流程监控。

在地方层面，各省市纷纷开发了本地区的物资管理系统。例如，上海市开发了公共卫生应急物资管理信息系统，实现了物资需求的在线申报、审批和调拨；浙江省建立了应急物资管理平台，整合了政府部门、医疗机构和社会力量的物资资源；深圳市开发了医疗物资智慧管理平台，运用物联网技术实现了物资的智能仓储和配送。

在学术研究方面，国内学者对应急物资管理问题进行了广泛研究。李明等（2021）提出了基于大数据的应急物资需求预测模型；王华等（2022）研究了基于区块链的应急物资溯源机制；张伟等（2023）设计了基于微服务架构的应急物资调度系统架构。这些研究为物资管理系统的开发提供了理论支撑。

从技术应用来看，国内物资管理系统普遍采用了B/S架构、关系型数据库和Web开发技术。近年来，随着微服务架构、云计算和大数据技术的普及，越来越多的系统开始采用这些新技术。然而，现有的系统在以下方面仍存在不足：一是系统的集成度不高，各子系统之间往往相互独立，数据共享困难；二是高并发处理能力不足，在突发情况下容易出现系统崩溃或响应缓慢的问题；三是智能化程度不高，缺乏科学的需求预测和优化调度算法。

#### 1.3.3 研究现状总结

综合国内外研究现状可以看出，公共卫生物资管理信息化建设已成为普遍共识，各种新技术在物资管理领域的应用日益广泛。然而，现有的研究和实践仍存在一些不足：一是缺乏专门针对疫情防控场景的物资调度管理系统；二是在高并发处理、数据一致性保障等方面还需要进一步研究；三是系统的智能化水平有待提高。本课题正是在这样的背景下，针对现有研究的不足，设计和开发一套基于Spring Cloud的疫情防控物资调度管理系统。

## 第2章 相关技术介绍

### 2.1 Spring Boot框架

#### 2.1.1 Spring Boot概述

Spring Boot是由Pivotal团队提供的基于Spring框架的开源项目，其设计目的是简化Spring应用的初始搭建以及开发过程。Spring Boot采用了"约定优于配置"的设计理念，通过提供默认配置和自动配置功能，大大减少了开发人员需要编写的配置文件数量，使开发者能够快速构建独立运行的、生产级别的Spring应用。

Spring Boot具有以下主要特点：

**（1）独立运行**：Spring Boot应用可以打包成可执行的JAR包或WAR包，内嵌Tomcat、Jetty或Undertow等Servlet容器，无需部署到外部容器中即可独立运行。

**（2）自动配置**：Spring Boot根据项目中引入的依赖自动配置Spring应用，开发者可以通过配置文件覆盖默认配置。

**（3）起步依赖**：Spring Boot提供了一系列"starter"依赖，简化了Maven或Gradle配置，开发者只需引入相应的starter即可获得所需的功能。

**（4）监控管理**：Spring Boot Actuator提供了生产级别的监控和管理功能，可以查看应用的运行状态、度量指标、健康信息等。

#### 2.1.2 Spring Boot核心原理

Spring Boot的自动配置机制是其核心特性之一。自动配置通过<@EnableAutoConfiguration>注解实现，Spring Boot会在启动时扫描类路径下的所有jar包，根据其中定义的<META-INF/spring.factories>文件加载自动配置类。每个自动配置类都会根据类路径中是否存在特定类、配置文件中是否定义了特定属性等条件，决定是否应用该配置。

Spring Boot的启动流程主要包括以下步骤：

1. 加载Spring Boot启动类，创建SpringApplication实例
2. 推断应用类型（Servlet、Reactive或None）
3. 加载ApplicationContextInitializer和ApplicationListener
4. 推断主类
5. 调用run方法启动应用
6. 准备Environment环境
7. 创建ApplicationContext应用上下文
8. 准备ApplicationContext
9. 刷新ApplicationContext，加载Bean定义
10. 执行Runner接口的实现类
11. 发布应用启动完成事件

在本系统中，Spring Boot作为主要的后端开发框架，负责处理业务逻辑、数据访问、接口暴露等功能。系统采用Spring Boot 3.2.2版本，基于Spring Framework 6.x和Java 17构建，充分利用了虚拟线程等新特性，提高了系统的并发处理能力。

### 2.2 Spring Cloud Alibaba

#### 2.2.1 微服务架构概述

微服务架构是一种将单一应用程序开发为一组小型服务的方法，每个服务运行在自己的进程中，服务之间通过轻量级机制（通常是HTTP API）进行通信。微服务架构具有以下优点：

**（1）服务独立**：每个微服务都是独立的，可以独立开发、部署和扩展，互不影响。

**（2）技术多样性**：不同的微服务可以采用不同的技术栈，选择最适合业务需求的技术。

**（3）弹性扩展**：可以根据实际需求对特定的服务进行水平扩展，提高资源利用效率。

**（4）故障隔离**：单个服务的故障不会导致整个系统的崩溃，提高了系统的可用性。

然而，微服务架构也带来了新的挑战，包括服务间的通信、服务发现、负载均衡、分布式事务、配置管理、熔断降级等问题。

#### 2.2.2 Spring Cloud Alibaba介绍

Spring Cloud Alibaba是阿里巴巴开源的微服务解决方案，是Spring Cloud的官方孵化器项目，于2018年正式加入Spring Cloud生态。Spring Cloud Alibaba提供了一站式微服务开发工具集，主要包括以下组件：

**（1）Nacos**：动态服务发现、配置管理和服务管理平台。Nacos支持服务注册与发现、动态配置推送、服务健康检查等功能，可以作为服务注册中心和配置中心使用。

**（2）Sentinel**：面向分布式服务架构的流量控制组件，提供了流量控制、熔断降级、系统负载保护等功能，保障微服务的稳定性。

**（3）Seata**：开源的分布式事务解决方案，提供了AT、TCC、SAGA和XA等多种事务模式，解决微服务架构下的数据一致性问题。

**（4）RocketMQ**：高性能、高可靠的分布式消息中间件，支持发布订阅、顺序消息、延时消息、事务消息等多种消息类型。

**（5）Dubbo**：高性能的RPC框架，支持多种协议和负载均衡策略，适用于微服务间的同步调用。

**（6）Alibaba Cloud OSS/SchedulerX**：提供对象存储和分布式任务调度等云服务能力。

#### 2.2.3 Spring Cloud Alibaba在系统中的应用

本系统采用Spring Cloud Alibaba 2023.0.1.0作为微服务框架，各服务的职责划分如下：

**网关服务（端口8080）**：基于Spring Cloud Gateway实现，负责统一接收和处理客户端请求，进行路由转发、负载均衡、认证鉴权、限流熔断等操作。网关服务是系统的统一入口，对外隐藏了内部微服务的实现细节。

**用户服务（端口8081）**：负责用户相关的业务逻辑，包括用户注册、登录认证、角色权限管理、用户信息维护等功能。用户服务采用JWT+Redis的双重验证机制，保障身份认证的安全性。

**物资服务（端口8082）**：负责物资相关的业务逻辑，包括物资信息管理、物资类型管理、仓库管理、库存管理等功能。物资服务使用Redisson分布式锁保障高并发场景下库存操作的数据一致性。

**疫情服务（端口8083）**：负责疫情相关的业务逻辑，包括疫情新闻发布、知识库管理、数据统计分析、消息推送等功能。疫情服务采用RabbitMQ实现异步消息处理，提高系统的响应性能。

各服务之间通过Feign进行声明式HTTP调用，通过Nacos进行服务注册与发现，通过Sentinel实现流量控制和熔断保护。

### 2.3 JWT认证技术

#### 2.3.1 JWT概述

JWT（JSON Web Token）是一种开放标准（RFC 7519），用于在网络应用环境间安全地将信息作为JSON对象传输。JWT可以用于身份认证和信息交换，具有体积小、易于传输、自包含等特点。

一个JWT由三部分组成，用点号分隔：

**Header（头部）**：包含令牌类型和签名算法信息，通常采用Base64Url编码。
```json
{
  "alg": "HS256",
  "typ": "JWT"
}
```

**Payload（载荷）**：包含声明（claims），声明是关于实体（通常是用户）和其他数据的声明。声明分为三种类型：注册声明（如iss、exp、sub等）、公共声明和私有声明。
```json
{
  "sub": "1234567890",
  "name": "John Doe",
  "iat": 1516239022
}
```

**Signature（签名）**：用于验证消息在此过程中未被更改，并且对于使用私钥签名的令牌，还可以验证JWT的发送方是否为它所声称的发送方。签名由编码后的头部、编码后的载荷、密钥和指定的算法生成。

#### 2.3.2 JWT工作流程

JWT的认证流程如下：

1. 用户使用用户名和密码登录服务器
2. 服务器验证用户信息，生成JWT令牌返回给客户端
3. 客户端将JWT令牌存储在本地（如localStorage或cookie）
4. 客户端发起后续请求时，在HTTP Header中携带JWT令牌
5. 服务器验证JWT令牌的签名和有效期
6. 验证通过后，服务器处理请求并返回响应

#### 2.3.3 JWT在系统中的应用

本系统采用JWT+Redis的双重验证机制。用户登录成功后，服务端生成JWT令牌并存储在Redis中，同时返回给客户端。客户端在后续请求中携带JWT令牌，服务端首先验证JWT的签名和有效期，然后查询Redis验证令牌是否有效。这种双重验证机制具有以下优点：

**（1）安全性高**：即使JWT令牌被窃取，攻击者也无法伪造有效的签名；同时服务端可以通过Redis随时使令牌失效。

**（2）支持单点登录**：多个系统可以共享同一个Redis，实现跨系统的单点登录。

**（3）可控性强**：服务端可以通过Redis控制令牌的有效期，支持主动注销功能。

### 2.4 Redis缓存技术

#### 2.4.1 Redis概述

Redis（Remote Dictionary Server）是一个开源的、基于内存的高性能键值对存储系统。Redis支持多种数据结构，包括字符串（String）、列表（List）、集合（Set）、有序集合（Sorted Set）、哈希（Hash）、位图（Bitmap）、HyperLogLog、地理空间索引（Geospatial）和流（Stream）等。Redis具有以下特点：

**（1）高性能**：Redis将数据存储在内存中，读写速度极快，可以达到每秒10万次的读写操作。

**（2）持久化**：Redis支持RDB和AOF两种持久化方式，可以将内存中的数据保存到磁盘，防止数据丢失。

**（3）高可用**：Redis支持主从复制、哨兵模式和集群模式，可以实现数据的高可用和自动故障转移。

**（4）丰富的功能**：Redis支持事务、Lua脚本、发布订阅、管道、慢查询分析等高级功能。

#### 2.4.2 Redis数据类型及应用场景

Redis支持的数据类型及其典型应用场景如下：

**String（字符串）**：最基本的数据类型，可以存储字符串、整数或浮点数。应用场景包括缓存、计数器、分布式锁等。

**Hash（哈希）**：适合存储对象，一个key可以存储多个field-value对。应用场景包括用户信息、商品信息、配置信息等。

**List（列表）**：有序的字符串列表，支持从两端进行插入和弹出操作。应用场景包括消息队列、时间线、最新消息列表等。

**Set（集合）**：无序的唯一字符串集合。应用场景包括标签系统、共同好友、抽奖活动等。

**Sorted Set（有序集合）**：类似于Set，但每个元素关联一个分数，用于排序。应用场景包括排行榜、延时队列等。

#### 2.4.3 Redisson分布式锁

Redisson是一个基于Redis的Java驻内存数据网格（In-Memory Data Grid），提供了许多分布式Java数据结构和服务，如分布式锁、分布式集合、分布式对象、分布式服务等。

Redisson分布式锁的主要特点包括：

**（1）可重入锁**：同一个线程可以多次获取同一把锁，不会发生死锁。

**（2）自动续期**：锁的持有时间可以自动续期，防止业务还未执行完锁就过期的情况。

**（3）看门狗机制**：Redisson会自动检测锁的持有情况，在锁即将过期时自动续期。

**（4）公平锁和非公平锁**：支持公平锁（按请求顺序获取锁）和非公平锁（允许插队）。

在本系统中，Redisson分布式锁用于保障库存操作的原子性，防止高并发场景下的超卖或库存不一致问题。

#### 2.4.4 Redis在系统中的应用

本系统中Redis主要用于以下几个方面：

**（1）会话缓存**：存储用户的JWT令牌和登录状态，实现分布式会话管理。

**（2）数据缓存**：缓存热点数据，如物资信息、用户信息、配置信息等，减少数据库访问压力。

**（3）分布式锁**：使用Redisson实现分布式锁，保障库存操作的原子性。

**（4）限流计数器**：基于Redis实现接口限流，防止恶意请求和突发流量。

**（5）排行榜**：使用Sorted Set实现物资消耗排行榜、用户活跃度排行榜等。

### 2.5 RabbitMQ消息队列

#### 2.5.1 消息队列概述

消息队列（Message Queue）是一种应用程序之间的通信方法，应用程序通过读写出入队列的消息来通信，无需专用连接来链接它们。消息队列提供了异步通信机制，将消息发送者和接收者解耦。

消息队列的主要应用场景包括：

**（1）异步处理**：将非核心业务流程异步化，提高系统响应速度。

**（2）应用解耦**：通过消息队列作为中介，降低系统组件之间的耦合度。

**（3）流量削峰**：在高并发场景下，使用消息队列缓冲请求，平滑处理峰值流量。

**（4）日志处理**：将日志信息发送到消息队列，异步写入存储系统。

**（5）延时任务**：实现延时执行的任务，如定时提醒、订单超时处理等。

#### 2.5.2 RabbitMQ介绍

RabbitMQ是一个开源的消息代理软件（消息中间件），实现了高级消息队列协议（AMQP）。RabbitMQ使用Erlang语言编写，具有高性能、高可靠、灵活路由等特点。

RabbitMQ的核心概念包括：

**Producer（生产者）**：创建消息并发送到交换机（Exchange）的应用程序。

**Consumer（消费者）**：连接到消息队列并获取消息的应用程序。

**Queue（队列）**：存储消息的缓冲区，消费者从队列中获取消息。

**Exchange（交换机）**：接收生产者发送的消息，并根据路由规则将消息路由到相应的队列。常用的交换机类型包括：
- direct：精确匹配路由键
- fanout：广播到所有绑定的队列
- topic：模式匹配路由键
- headers：根据消息头属性匹配

**Binding（绑定）**：交换机和队列之间的关联关系，定义了消息从交换机到队列的路由规则。

#### 2.5.3 RabbitMQ在系统中的应用

本系统中RabbitMQ主要用于以下几个方面：

**（1）异步日志处理**：系统操作日志通过RabbitMQ异步写入数据库，避免日志记录影响主业务流程的性能。

**（2）消息通知**：系统通知、审批提醒等消息通过RabbitMQ异步发送，支持邮件、短信、站内信等多种通知方式。

**（3）数据同步**：各服务之间的数据同步通过RabbitMQ实现，保证数据的最终一致性。

**（4）延时任务**：使用RabbitMQ的死信队列实现延时任务，如库存预警检查、订单超时处理等。

### 2.7 MyBatis-Plus持久层框架

#### 2.7.1 MyBatis概述

MyBatis是一款优秀的持久层框架，它支持定制化SQL、存储过程以及高级映射。MyBatis避免了几乎所有的JDBC代码和手动设置参数以及获取结果集。MyBatis可以使用简单的XML或注解来配置和映射原生信息，将接口和Java的POJOs（Plain Old Java Objects）映射成数据库中的记录。

MyBatis-Plus（简称MP）是一个MyBatis的增强工具，在MyBatis的基础上只做增强不做改变，为简化开发、提高效率而生。MyBatis-Plus提供了以下核心功能：

**（4）分页插件**：内置了物理分页插件，支持多种数据库，使用简单。

**（5）性能分析插件**：可以输出SQL语句及其执行时间，方便进行性能优化。

**（6）多租户插件**：支持多租户数据隔离，可以自动添加租户ID过滤条件。

**（7）动态表名插件**：支持动态切换表名，适用于分表场景。

#### 2.7.3 MyBatis-Plus在系统中的应用

### 3.2 业务需求分析

#### 3.2.1 业务流程分析

疫情防控物资调度的业务流程主要包括以下几个环节：

**（1）物资采购与入库**

物资管理部门根据库存情况和需求预测，制定采购计划并采购物资。物资到货后，进行入库登记，更新库存数量。入库信息包括物资名称、规格型号、数量、生产厂家、生产日期、有效期、存放位置等。

**（2）库存管理**

仓库管理员定期对库存进行盘点，核对实际库存与系统记录是否一致。系统根据设定的库存预警阈值，自动发出库存预警提醒。当物资即将过期时，系统也会发出预警提醒。

**（3）物资申请**

医疗机构或社区工作人员根据实际需求，通过系统提交物资申请。申请信息包括申请单位、申请物资、申请数量、申请理由、紧急程度等。申请提交后，进入审批流程。

**（4）申请审批**

申请按照设定的审批流程逐级审批。审批人可以查看申请的详细信息，根据实际情况批准或驳回申请。审批过程中可以添加审批意见，申请被驳回后可以修改后重新提交。

**（5）物资调配**

申请审批通过后，生成调配单。仓库管理员根据调配单进行物资出库操作，记录出库信息。物资配送到申请单位后，申请单位确认收货，完成整个调配流程。

- 支持记住密码功能
- 支持密码找回功能

- 查看个人基本信息
- 修改个人资料（姓名、手机号、邮箱等）

- 机构管理：维护医疗机构、社区等单位信息

#### 3.3.3 库存管理模块

**（1）库存查询**

- 实时库存查询：查看各仓库、各物资的当前库存数量
- 库存明细查询：查看库存的批次、有效期、存放位置等详细信息
- 库存流水查询：查看库存的出入库历史记录
- 多条件查询：支持按物资、仓库、时间段等条件查询

**（2）入库管理**

- 采购入库：根据采购订单进行入库
- 捐赠入库：根据捐赠记录进行入库
- 退库入库：处理退货入库
- 入库单管理：创建、审核、查询入库单
- 入库明细：记录入库物资的批次、数量、单价、有效期等

**（3）出库管理**

- 申请出库：根据批准的物资申请进行出库
- 调拨出库：处理仓库间的物资调拨
- 损耗出库：处理物资损耗、过期等出库
- 出库单管理：创建、审核、查询出库单
- 出库明细：记录出库物资的批次、数量、去向等

**（4）库存预警**

- 预警规则设置：设置各类物资的安全库存和最高库存
- 低库存预警：当库存低于安全库存时自动预警
- 高库存预警：当库存高于最高库存时提醒
- 效期预警：当物资即将过期时提前预警
- 预警通知：通过系统消息、短信、邮件等方式通知相关人员

**（5）库存盘点**

- 盘点单创建：创建盘点任务，指定盘点范围和盘点人员
- 盘点执行：扫描或录入实际盘点数量
- 盘点差异处理：处理盘点中发现的库存差异
- 盘点报告：生成盘点报告，统计盘盈盘亏情况

#### 3.3.4 申请管理模块

申请管理模块是物资调度的核心流程，主要功能包括：

**（1）物资申请**

- 申请单创建：选择需要申请的物资，填写申请数量、申请理由、紧急程度等
- 申请单暂存：支持保存为草稿，后续继续编辑
- 申请单提交：提交后进入审批流程
- 申请单撤回：在审批前可以撤回申请

**（2）申请审批**

- 审批流程配置：配置不同类型申请的审批流程
- 待审批列表：审批人查看待审批的申请
- 审批操作：批准、驳回、转办等操作
- 审批意见：填写审批意见
- 审批历史：查看申请的审批历史记录

**（3）申请查询**

- 我的申请：申请人查看自己提交的申请
- 申请状态查询：按待提交、审批中、已通过、已驳回等状态查询
- 申请详情：查看申请的详细信息和审批进度

**（4）物流配送**

- 配送单生成：申请审批通过后自动生成配送单
- 配送状态跟踪：记录配送的各个节点状态
- 收货确认：申请单位确认收货
- 物流轨迹查询：查询物资的配送轨迹

### 3.4 非功能需求分析

#### 3.4.1 性能需求

**响应时间**：
- 页面加载时间不超过3秒
- 简单查询响应时间不超过1秒
- 复杂查询响应时间不超过3秒
- 提交操作响应时间不超过2秒

**并发能力**：
- 系统支持至少1000个并发用户
- 关键接口支持至少500TPS（每秒事务数）
- 库存操作支持高并发，保证数据一致性

**可用性**：
- 系统可用性不低于99.9%
- 支持负载均衡，单点故障不影响整体服务
- 支持服务熔断和降级，故障快速恢复

#### 3.4.2 安全需求

**身份认证**：
- 用户必须通过身份认证才能访问系统
- 采用JWT+Redis双重验证机制
- 密码采用加密存储，支持密码强度校验
- 支持登录失败次数限制和账户锁定

**权限控制**：
- 基于RBAC模型进行权限控制
- 支持细粒度的操作权限控制
- 支持数据权限控制，用户只能访问权限范围内的数据

**数据安全**：
- 敏感数据传输采用HTTPS加密
- 敏感数据存储采用加密算法
- 数据修改记录审计日志
- 支持数据备份和恢复

**防护机制**：
- 防止SQL注入攻击
- 防止XSS跨站脚本攻击
- 防止CSRF跨站请求伪造攻击
- 接口限流，防止暴力破解和DDoS攻击

#### 3.4.3 可靠性需求

**数据一致性**：
- 采用分布式事务保证跨服务数据一致性
- 库存操作保证原子性，防止超卖
- 重要操作支持幂等性，防止重复提交

**容错性**：
- 服务故障自动切换，不影响业务运行
- 数据库主从切换，保证数据服务连续性
- 消息队列保证消息不丢失

**可恢复性**：
- 支持数据定期备份
- 支持系统快速恢复
- 支持灾难恢复

#### 3.4.4 易用性需求

**界面友好**：
- 界面布局清晰，符合用户使用习惯
- 操作流程简洁，减少用户操作步骤
- 提供操作提示和帮助信息

**响应式设计**：
- 支持不同分辨率的屏幕
- 支持主流浏览器

**帮助文档**：
- 提供用户操作手册
- 提供在线帮助信息
- 提供常见问题解答

#### 3.4.5 可维护性需求

**代码规范**：
- 遵循统一的编码规范
- 代码结构清晰，易于理解
- 代码注释完整，便于维护

**日志记录**：
- 记录系统运行日志
- 记录用户操作日志
- 记录异常错误日志
- 支持日志查询和分析

**监控告警**：
- 系统运行状态监控
- 服务健康检查
- 异常情况自动告警

**配置管理**：
- 支持动态配置更新
- 配置集中管理
- 支持灰度发布

### 3.5 本章小结

本章对疫情防控物资调度管理系统进行了全面的需求分析。首先从技术、经济、操作和法律四个方面进行了可行性分析，论证了系统开发的可行性。然后对业务流程进行了详细分析，明确了系统的业务参与者。接着对系统功能需求进行了详细描述，包括用户管理、物资管理、库存管理、申请管理、捐赠管理、疫情信息和数据统计七大功能模块。最后对系统的非功能需求进行了分析，包括性能需求、安全需求、可靠性需求、易用性需求和可维护性需求。本章的需求分析为后续的系统设计提供了明确的依据。

#### 4.1.2 微服务划分

根据业务领域边界，系统划分为以下微服务：

**网关服务（Gateway Service）**：端口8080，基于Spring Cloud Gateway，负责统一接收和处理客户端请求、路由转发、负载均衡、认证鉴权、限流熔断等。

**用户服务（User Service）**：端口8081，数据库user_db，负责用户注册登录、用户信息管理、角色权限管理、组织机构管理等。

**物资服务（Material Service）**：端口8082，数据库material_db，负责物资类型管理、物资信息管理、仓库管理、库存管理、库存预警、物资申请管理、物资捐赠管理等。

**疫情服务（Epidemic Service）**：端口8083，数据库epidemic_db，负责疫情新闻发布、知识库管理、数据统计分析、消息推送等。

#### 4.1.3 技术架构选型

**服务注册与发现**：采用Nacos作为服务注册中心和配置中心，支持AP和CP两种模式。

**服务调用**：采用OpenFeign + Ribbon实现声明式HTTP调用和客户端负载均衡。

**服务熔断与限流**：采用Sentinel实现流量控制、熔断降级和系统保护。

**分布式事务**：采用Seata的AT模式实现分布式事务，通过解析SQL生成反向SQL实现事务回滚。

**分布式锁**：采用Redisson实现分布式锁，支持可重入锁、看门狗自动续期等特性。

**缓存策略**：采用多级缓存，一级缓存Caffeine本地缓存，二级缓存Redis分布式缓存，三级缓存数据库。

**消息队列**：采用RabbitMQ作为消息中间件，用于异步处理、应用解耦、流量削峰等场景。

**链路追踪**：采用SkyWalking实现分布式链路追踪。

本章对疫情防控物资调度管理系统进行了详细的设计。首先设计了系统的总体架构，包括接入层、网关层、服务层、数据层和基础设施层。然后对系统的四个微服务进行了详细的功能划分。接着对各功能模块进行了详细设计，包括用户管理、物资管理、库存管理、申请管理和捐赠管理模块。之后进行了数据库设计，包括数据库选型、架构设计和各服务的具体表结构设计。最后设计了系统的接口规范和各服务的API接口。

```java
/**
 * 入库操作实现（使用分布式锁保障原子性）
 * @param dto 入库参数
 * @return 入库单号
 */
@Override
@Transactional(rollbackFor = Exception.class)
public String inbound(InboundDTO dto) {
    // 1. 参数校验
    if (dto.getQuantity() <= 0) {
        throw new BusinessException("入库数量必须大于0");
    }
    
    // 2. 生成入库单号
    String orderNo = generateOrderNo("IN");
    
    // 3. 获取分布式锁（防止并发操作同一物资库存）
    String lockKey = "inventory:lock:" + dto.getMaterialId() + ":" + dto.getWarehouseId();
    RLock lock = redissonClient.getLock(lockKey);
    
    try {
        // 尝试获取锁，最多等待10秒，锁自动释放时间30秒
        boolean acquired = lock.tryLock(10, 30, TimeUnit.SECONDS);
        if (!acquired) {
            throw new BusinessException("系统繁忙，请稍后重试");
        }
        
        try {
            // 4. 查询当前库存
            Inventory inventory = inventoryMapper.selectByMaterialAndWarehouse(
                dto.getMaterialId(), dto.getWarehouseId());
            
            int beforeQuantity = 0;
            if (inventory == null) {
                // 新建库存记录
                inventory = new Inventory();
                inventory.setMaterialId(dto.getMaterialId());
                inventory.setWarehouseId(dto.getWarehouseId());
                inventory.setBatchNo(dto.getBatchNo());
                inventory.setQuantity(dto.getQuantity());
                inventory.setLockedQuantity(0);
                inventory.setAvailableQuantity(dto.getQuantity());
                inventory.setUnitCost(dto.getUnitCost());
                inventory.setProductionDate(dto.getProductionDate());
                inventory.setExpiryDate(dto.getExpiryDate());
                inventoryMapper.insert(inventory);
            } else {
                // 更新库存数量
                beforeQuantity = inventory.getQuantity();
                inventory.setQuantity(inventory.getQuantity() + dto.getQuantity());
                inventory.setAvailableQuantity(inventory.getAvailableQuantity() + dto.getQuantity());
                // 更新单位成本（移动加权平均）
                if (dto.getUnitCost() != null) {
                    BigDecimal newCost = calculateAverageCost(inventory, dto);
                    inventory.setUnitCost(newCost);
                }
                inventoryMapper.updateById(inventory);
            }
            
            // 5. 创建入库单
            StockOrder order = new StockOrder();
            order.setOrderNo(orderNo);
            order.setOrderType("inbound");
            order.setMaterialId(dto.getMaterialId());
            order.setWarehouseId(dto.getWarehouseId());
            order.setQuantity(dto.getQuantity());
            order.setUnitCost(dto.getUnitCost());
            order.setStatus(1); // 已完成
            order.setCreateBy(UserContext.getUserId());
            stockOrderMapper.insert(order);
            
            // 6. 记录库存流水
            InventoryLog log = new InventoryLog();
            log.setMaterialId(dto.getMaterialId());
            log.setWarehouseId(dto.getWarehouseId());
            log.setChangeType("inbound");
            log.setChangeQuantity(dto.getQuantity());
            log.setBeforeQuantity(beforeQuantity);
            log.setAfterQuantity(inventory.getQuantity());
            log.setOrderNo(orderNo);
            log.setRemark(dto.getRemark());
            log.setCreateBy(UserContext.getUserId());
            inventoryLogMapper.insert(log);
            
            // 7. 异步发送库存变更消息（用于预警检查）
            rabbitTemplate.convertAndSend(
                "inventory.exchange", 
                "inventory.change", 
                new InventoryChangeEvent(dto.getMaterialId(), dto.getWarehouseId())
            );
            
            return orderNo;
        } finally {
            // 释放锁
            lock.unlock();
        }
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();
        throw new BusinessException("操作被中断");
    }
}

/**
 * 计算移动加权平均成本
 */
private BigDecimal calculateAverageCost(Inventory inventory, InboundDTO dto) {
    BigDecimal totalCost = inventory.getUnitCost()
        .multiply(BigDecimal.valueOf(inventory.getQuantity()))
        .add(dto.getUnitCost().multiply(BigDecimal.valueOf(dto.getQuantity())));
    BigDecimal totalQuantity = BigDecimal.valueOf(
        inventory.getQuantity() + dto.getQuantity());
    return totalCost.divide(totalQuantity, 2, RoundingMode.HALF_UP);
}
```

**库存预警消费者**：监听库存变更消息，检查并发送预警，采用幂等性控制防止重复处理。

```java
/**
 * 库存变更消息消费者
 */
@Component
@Slf4j
public class InventoryChangeConsumer {
    
    @Autowired
    private MaterialService materialService;
    
    @Autowired
    private InventoryAlertService alertService;
    
    @Autowired
    private NotificationService notificationService;
    
    @Autowired
    private StringRedisTemplate redisTemplate;
    
    /**
     * 监听库存变更消息，检查并触发预警
     */
    @RabbitListener(queues = "inventory.change.queue")
    public void handleInventoryChange(InventoryChangeEvent event) {
        log.info("收到库存变更消息: materialId={}, warehouseId={}", 
            event.getMaterialId(), event.getWarehouseId());
        
        try {
            // 幂等性控制：使用消息ID去重
            String messageId = event.getMessageId();
            String key = "inventory:change:processed:" + messageId;
            Boolean setIfAbsent = redisTemplate.opsForValue()
                .setIfAbsent(key, "1", 1, TimeUnit.HOURS);
            if (Boolean.FALSE.equals(setIfAbsent)) {
                log.info("消息已处理，跳过: messageId={}", messageId);
                return;
            }
            
            // 查询物资信息
            Material material = materialService.getById(event.getMaterialId());
            if (material == null) {
                log.warn("物资不存在: materialId={}", event.getMaterialId());
                return;
            }
            
            // 查询库存信息
            Inventory inventory = inventoryService.getByMaterialAndWarehouse(
                event.getMaterialId(), event.getWarehouseId());
            if (inventory == null) {
                return;
            }
            
            // 检查库存预警
            if (inventory.getQuantity() <= material.getThreshold()) {
                // 创建或更新预警记录
                InventoryAlert alert = new InventoryAlert();
                alert.setMaterialId(event.getMaterialId());
                alert.setWarehouseId(event.getWarehouseId());
                alert.setCurrentQuantity(inventory.getQuantity());
                alert.setThreshold(material.getThreshold());
                alert.setAlertLevel(calculateAlertLevel(inventory.getQuantity(), material.getThreshold()));
                alert.setStatus(0); // 未处理
                alertService.saveOrUpdate(alert);
                
                // 发送预警通知
                notificationService.sendAlertNotification(material, inventory);
            } else {
                // 库存恢复正常，清除预警记录
                alertService.clearAlert(event.getMaterialId(), event.getWarehouseId());
            }
            
            // 检查有效期预警
            if (inventory.getExpiryDate() != null) {
                long daysToExpiry = ChronoUnit.DAYS.between(
                    LocalDate.now(), inventory.getExpiryDate());
                if (daysToExpiry <= 30) {
                    // 发送即将过期预警
                    notificationService.sendExpiryAlert(material, inventory, daysToExpiry);
                }
            }
            
        } catch (Exception e) {
            log.error("处理库存变更消息失败", e);
            // 消息处理失败，抛出异常触发重试
            throw new AmqpRejectAndDontRequeueException(e);
        }
    }
    
    /**
     * 计算预警级别
     */
    private String calculateAlertLevel(Integer quantity, Integer threshold) {
        double ratio = (double) quantity / threshold;
        if (ratio <= 0.3) {
            return "高";
        } else if (ratio <= 0.5) {
            return "中";
        } else {
            return "低";
        }
    }
}
```

#### 5.7.1 JWT工具类实现

包括生成AccessToken、生成RefreshToken、从Token获取用户ID、解析Token、验证Token有效性等方法。

```java
/**
 * JWT工具类
 */
@Component
@Slf4j
public class JwtUtil {
    
    @Value("${jwt.secret}")
    private String secret;
    
    @Value("${jwt.expiration:1800000}") // 默认30分钟
    private Long expiration;
    
    @Value("${jwt.refresh-expiration:604800000}") // 默认7天
    private Long refreshExpiration;
    
    private SecretKey key;
    
    @PostConstruct
    public void init() {
        // 使用HS256算法生成密钥
        key = Keys.hmacShaKeyFor(secret.getBytes(StandardCharsets.UTF_8));
    }
    
    /**
     * 生成访问令牌
     * @param userId 用户ID
     * @param username 用户名
     * @param roles 角色列表
     * @return JWT Token
     */
    public String generateAccessToken(Long userId, String username, List<String> roles) {
        Date now = new Date();
        Date expiryDate = new Date(now.getTime() + expiration);
        
        return Jwts.builder()
            .subject(String.valueOf(userId))
            .claim("username", username)
            .claim("roles", roles)
            .claim("type", "access")
            .issuedAt(now)
            .expiration(expiryDate)
            .signWith(key, Jwts.SIG.HS256)
            .compact();
    }
    
    /**
     * 生成刷新令牌
     * @param userId 用户ID
     * @return Refresh Token
     */
    public String generateRefreshToken(Long userId) {
        Date now = new Date();
        Date expiryDate = new Date(now.getTime() + refreshExpiration);
        
        return Jwts.builder()
            .subject(String.valueOf(userId))
            .claim("type", "refresh")
            .claim("random", UUID.randomUUID().toString())
            .issuedAt(now)
            .expiration(expiryDate)
            .signWith(key, Jwts.SIG.HS256)
            .compact();
    }
    
    /**
     * 从Token中获取用户ID
     * @param token JWT Token
     * @return 用户ID
     */
    public Long getUserId(String token) {
        Claims claims = parseToken(token);
        return Long.parseLong(claims.getSubject());
    }
    
    /**
     * 从Token中获取用户名
     * @param token JWT Token
     * @return 用户名
     */
    public String getUsername(String token) {
        Claims claims = parseToken(token);
        return claims.get("username", String.class);
    }
    
    /**
     * 从Token中获取角色列表
     * @param token JWT Token
     * @return 角色列表
     */
    @SuppressWarnings("unchecked")
    public List<String> getRoles(String token) {
        Claims claims = parseToken(token);
        return claims.get("roles", List.class);
    }
    
    /**
     * 解析JWT Token
     * @param token JWT Token
     * @return Claims
     */
    public Claims parseToken(String token) {
        try {
            return Jwts.parser()
                .verifyWith(key)
                .build()
                .parseSignedClaims(token)
                .getPayload();
        } catch (ExpiredJwtException e) {
            log.warn("JWT Token已过期: {}", e.getMessage());
            throw new BusinessException("Token已过期，请重新登录");
        } catch (UnsupportedJwtException e) {
            log.warn("不支持的JWT Token: {}", e.getMessage());
            throw new BusinessException("无效的Token");
        } catch (MalformedJwtException e) {
            log.warn("JWT Token格式错误: {}", e.getMessage());
            throw new BusinessException("无效的Token");
        } catch (SignatureException e) {
            log.warn("JWT Token签名验证失败: {}", e.getMessage());
            throw new BusinessException("无效的Token");
        } catch (Exception e) {
            log.error("JWT Token解析失败: {}", e.getMessage());
            throw new BusinessException("Token验证失败");
        }
    }
    
    /**
     * 验证Token是否有效
     * @param token JWT Token
     * @return 是否有效
     */
    public boolean validateToken(String token) {
        try {
            parseToken(token);
            return true;
        } catch (Exception e) {
            return false;
        }
    }
    
    /**
     * 检查Token是否即将过期（小于5分钟）
     * @param token JWT Token
     * @return 是否即将过期
     */
    public boolean isTokenExpiringSoon(String token) {
        try {
            Claims claims = parseToken(token);
            Date expiration = claims.getExpiration();
            long timeLeft = expiration.getTime() - System.currentTimeMillis();
            return timeLeft < 5 * 60 * 1000; // 小于5分钟
        } catch (Exception e) {
            return true;
        }
    }
}
```

```java
/**
 * 分布式锁工具类
 */
@Component
@Slf4j
public class DistributedLockUtil {
    
    @Autowired
    private RedissonClient redissonClient;
    
    /**
     * 执行带锁的业务逻辑
     * @param lockKey 锁的key
     * @param waitTime 等待时间（秒）
     * @param leaseTime 持有时间（秒）
     * @param supplier 业务逻辑
     * @return 业务逻辑返回值
     */
    public <T> T executeWithLock(String lockKey, long waitTime, long leaseTime, 
                                  Supplier<T> supplier) {
        RLock lock = redissonClient.getLock(lockKey);
        try {
            // 尝试获取锁
            boolean acquired = lock.tryLock(waitTime, leaseTime, TimeUnit.SECONDS);
            if (!acquired) {
                throw new BusinessException("系统繁忙，请稍后重试");
            }
            
            try {
                // 执行业务逻辑
                return supplier.get();
            } finally {
                // 释放锁
                if (lock.isHeldByCurrentThread()) {
                    lock.unlock();
                }
            }
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            throw new BusinessException("操作被中断");
        }
    }
    
    /**
     * 执行带锁的业务逻辑（无返回值）
     */
    public void executeWithLock(String lockKey, long waitTime, long leaseTime, 
                                 Runnable runnable) {
        RLock lock = redissonClient.getLock(lockKey);
        try {
            boolean acquired = lock.tryLock(waitTime, leaseTime, TimeUnit.SECONDS);
            if (!acquired) {
                throw new BusinessException("系统繁忙，请稍后重试");
            }
            
            try {
                runnable.run();
            } finally {
                if (lock.isHeldByCurrentThread()) {
                    lock.unlock();
                }
            }
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            throw new BusinessException("操作被中断");
        }
    }
    
    /**
     * 带重试机制的分布式锁执行
     */
    public <T> T executeWithRetry(String lockKey, int maxRetries, Supplier<T> supplier) {
        int retryCount = 0;
        while (retryCount < maxRetries) {
            try {
                return executeWithLock(lockKey, 5, 30, supplier);
            } catch (BusinessException e) {
                if ("系统繁忙，请稍后重试".equals(e.getMessage()) && retryCount < maxRetries - 1) {
                    retryCount++;
                    try {
                        // 随机等待100-300ms后重试
                        Thread.sleep(100 + RandomUtil.randomInt(200));
                    } catch (InterruptedException ie) {
                        Thread.currentThread().interrupt();
                        throw new BusinessException("操作被中断");
                    }
                } else {
                    throw e;
                }
            }
        }
        throw new BusinessException("操作失败，请稍后重试");
    }
}
```
